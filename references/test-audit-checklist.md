# Test Audit Checklist

## Overview

The test audit validates that tests actually PROVE requirements are met, not just that they pass.

## 5 Sub-Checks

### 1. Disabled Test Scan

**Purpose:** Detect skipped/disabled tests that leave requirements untested.

**Search patterns:**
- JavaScript/TypeScript: `it.skip`, `describe.skip`, `test.skip`, `xit`, `xdescribe`, `xtest`
- Python: `@pytest.mark.skip`, `@unittest.skip`, `@skip`
- Rust: `#[ignore]`
- General: `.pending`, `it.todo`, `test.todo`

**Rule:** A disabled test linked to a requirement = requirement NOT tested.
**Severity:** BLOCKER if disabled test is the ONLY test proving that requirement.

### 2. Circular Test Detection

**Purpose:** Detect tests that generate their own expected values (self-referential).

**Search patterns:**
- `writeFileSync`, `writeFile`, `fs.write`, `open(.*w)` in test directories
- Script imports service AND writes to fixture files
- Expected values have comments: "computed from engine", "captured from baseline"
- Filename contains: "capture", "baseline", "generate", "snapshot"

**Rule:** A test comparing system output against values generated by the same system is circular. It proves consistency, not correctness.
**Severity:** BLOCKER

### 3. Assertion Strength Validation

**Purpose:** Ensure assertions are strong enough for the requirement.

**Strength levels (weakest to strongest):**
- **Existence:** `toBeDefined()`, `!= null` → something returned
- **Type:** `typeof x === 'number'` → correct shape
- **Status:** `code === 200` → no error
- **Value:** `toEqual(expected)`, `toBeCloseTo(x)` → specific value
- **Behavioral:** Multi-step workflow assertions → end-to-end correctness

**Rule:** If requirement demands value-level or behavioral-level proof and test only has existence/type/status assertions → INSUFFICIENT.
**Severity:** WARNING

### 4. Expected Value Provenance

**Purpose:** Validate that expected values come from external sources, not the system being tested.

**Provenance classifications:**
- **VALID:** Expected value from external/legacy system, manual capture, or independent oracle
- **PARTIAL:** Expected value from mathematical derivation (proves formula, not system match)
- **CIRCULAR:** Expected value from the system being tested
- **UNKNOWN:** No provenance information → treat as SUSPECT

**Rule:** For comparison/parity/migration requirements, external source must be invoked in test pipeline.
**Severity:** BLOCKER if CIRCULAR, WARNING if UNKNOWN

### 5. Coverage Quantity

**Purpose:** Verify test count meets requirement specifications.

**Check:** If requirement specifies quantity of test cases (e.g., "test 5 edge cases"), verify actual count meets requirement.
**Severity:** WARNING

## Verdict Calculation

- Any BLOCKER → overall test audit = BLOCKED
- All pass → overall test audit = PASSED
- Only WARNINGs → overall test audit = PASSED with warnings

## Integration with Verifier

The test-auditor agent runs after implementation and before verification. Its report is consumed by the verifier to determine overall verification status.
